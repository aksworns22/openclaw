## LangChain 출력 파싱과 체인 연결


<p><a href="2026-06-17.html">2026-06-17</a>에서 다룬 <code>ChatOllama</code>와 <code>PromptTemplate</code>을 기반으로, 모델 출력을 원하는 형태로 받아내고 여러 단계를 체인으로 엮는 방법을 실습했다.</p>

### StrOutputParser: AIMessage에서 문자열만 꺼내기


<p><code>llm.invoke()</code>의 결과는 <code>content</code> 외에 메타데이터를 함께 담은 <code>AIMessage</code> 객체다. <code>StrOutputParser</code>는 여기서 <code>content</code>만 뽑아 순수 문자열로 변환한다.</p>

<pre><code>from langchain_core.output_parsers import StrOutputParser

result = llm.invoke(prompt)          # AIMessage(content='Tokyo', ...)
output_parser = StrOutputParser()
parsed_result = output_parser.invoke(result)   # 'Tokyo'
</code></pre>

### JsonOutputParser: JSON 응답을 dict로


<p>프롬프트에서 "JSON 형식으로만 답하라"고 지시하면, 모델이 JSON 문자열을 생성한다. <code>JsonOutputParser</code>는 이를 파싱해 Python dict로 변환한다.</p>

<pre><code>from langchain_core.output_parsers import JsonOutputParser

country_detail_prompt = PromptTemplate(
    template="""Give following information about {country}:
    - Capital
    - Population
    - Language
    - Currency

    return it in JSON format, and return the JSON dictionary only
    """,
    input_variables=["country"],
)

json_output_parser = JsonOutputParser()
json_output_parser.invoke(llm.invoke(country_detail_prompt.invoke({'country': 'Japan'})))
# {'Capital': 'Tokyo', 'Population': 'Approx. 125 million', ...}
</code></pre>

<p>다만 이 방식은 모델이 형식 지시를 따르는 데 의존하므로, 필드 이름이나 자료형이 매번 달라질 수 있다.</p>

### with_structured_output: Pydantic 모델로 강제하기


<p>원하는 출력 스키마를 Pydantic <code>BaseModel</code>로 정의하고 <code>llm.with_structured_output(스키마)</code>로 감싸면, 모델이 그 스키마에 맞는 객체를 반환하도록 강제할 수 있다. 필드명과 타입이 고정되므로 <code>JsonOutputParser</code>보다 안정적이다.</p>

<pre><code>from pydantic import BaseModel, Field

class CountryDetail(BaseModel):
    capital: str = Field(description="The capital of the country")
    population: int = Field(description="The population of the country")
    language: str = Field(description="The language of the country")
    currency: str = Field(description="The currency of the country")

structured_llm = llm.with_structured_output(CountryDetail)
structured_llm.invoke(country_detail_prompt.invoke({'country': 'Japan'}))
# CountryDetail(capital='Tokyo', population=125990378, language='Japanese', currency='Japanese Yen')
</code></pre>

<p><code>population</code>이 정수형(<code>int</code>)으로 정확히 채워지는 점에서, 자유 형식 JSON 파싱보다 타입 안정성이 높다는 걸 확인할 수 있다.</p>

### <code>|</code> 연산자로 체인 만들기 (LCEL)


<p>LangChain은 <code>프롬프트 | 모델 | 파서</code>처럼 <code>|</code>로 컴포넌트를 이어 붙여 하나의 실행 가능한 체인(Runnable)을 만든다. 각 단계의 출력이 다음 단계의 입력으로 그대로 전달된다.</p>

<pre><code>country_chain = country_prompt | llm | output_parser
country_chain.invoke({'information': '에펠탑으로 유명한 나라'})
# '프랑스'
</code></pre>

### 체인을 다시 체인으로 합성하기


<p>체인 자체도 Runnable이므로, 체인과 체인을 다시 <code>|</code>로 연결할 수 있다.</p>

<pre><code>capital_chain = country_detail_prompt | llm | output_parser
final_chain = country_chain | capital_chain

final_chain.invoke({'information': '콜로세움으로 유명한 나라'})
</code></pre>

<p>여기서 주의할 점: <code>country_chain</code>의 출력(국가 이름 문자열)이 <code>capital_chain</code>의 입력으로 그대로 들어가는데, <code>capital_chain</code>이 기대하는 입력은 <code>{'country': ...}</code> 형태의 dict가 아니라 <code>country_detail_prompt</code>의 <code>input_variables</code>에 맞는 값이어야 한다. 실습에서는 문자열이 바로 들어가도 <code>PromptTemplate</code>이 단일 변수를 자동으로 채워주는 동작에 의존하고 있어, 두 체인의 입출력 타입이 자연스럽게 맞아떨어진 경우다.</p>

### RunnablePassthrough: 원본 입력을 그대로 다음 단계로


<p>체인 중간에서 원본 입력값을 가공하지 않고 그대로 다음 단계의 특정 키에 꽂아 넣고 싶을 때 <code>RunnablePassthrough</code>를 쓴다.</p>

<pre><code>from langchain_core.runnables import RunnablePassthrough

real_final_chain = (
    {'information': RunnablePassthrough()}
    | {'country': country_chain}
    | capital_chain
)

real_final_chain.invoke({'information': '김치의 나라'})
</code></pre>

<p>이 패턴은 "입력을 그대로 보존하면서, 동시에 그 입력을 가공한 결과도 함께 다음 단계에 넘기고 싶을 때" 자주 쓰인다. 여기서는 <code>{'information': ...}</code> 형태의 입력을 받아 <code>RunnablePassthrough</code>로 그대로 통과시키고, 그 값을 <code>country_chain</code>에 흘려보내 <code>{'country': ...}</code> 형태로 변환한 뒤 <code>capital_chain</code>에 전달하는 흐름이다.</p>

<div class="mermaid">
graph LR
  input["{'information': '김치의 나라'}"] --> pt["RunnablePassthrough"]
  pt --> cc["country_chain<br>(country_prompt | llm | parser)"]
  cc --> country["{'country': '한국'}"]
  country --> capital["capital_chain<br>(country_detail_prompt | llm | parser)"]
  capital --> result["최종 결과"]
</div>

## n8n으로 만든 LangChain Agent + Tool 워크플로


### 구조


<p>n8n의 <code>@n8n/n8n-nodes-langchain</code> 노드들을 이용해 채팅 기반 AI 에이전트를 구성했다. 노드 연결은 다음과 같다.</p>

<div class="mermaid">
graph TD
  trigger["Chat Trigger<br>(When chat message received)"] -->|main| agent["AI Agent"]
  model["Ollama Chat Model<br>(gemma4:e4b)"] -.->|ai_languageModel| agent
  tool["Code Tool<br>(덧셈)"] -.->|ai_tool| agent
</div>

<ul>
  <li><strong>Chat Trigger</strong>: 사용자가 채팅으로 입력하면 워크플로가 시작된다.</li>
  <li><strong>AI Agent</strong>: 실제 추론을 담당하는 LangChain 에이전트 노드. system message로 도구 호출 포맷을 지정해뒀다.</li>
  <li><strong>Ollama Chat Model</strong>: 로컬 Ollama에서 구동 중인 <code>gemma4:e4b</code> 모델을 LLM으로 연결.</li>
  <li><strong>Code Tool</strong>: 에이전트가 호출할 수 있는 커스텀 도구로, JS 코드로 숫자 배열의 합을 계산.</li>
</ul>

### Agent의 system message로 도구 호출 형식 강제하기


<pre><code>덧셈을 하기 위해서는 출력을 다음과 같이 만들어
ex) { "numbers": [1, 2, 3] }
</code></pre>

<p>모델이 자유 형식으로 답하는 대신, 도구(Code Tool)가 기대하는 입력 스키마(<code>{"numbers": [...]}</code>)에 맞춰 출력하도록 system message에서 명시적으로 예시를 줬다. 이건 아래 "Tool Calling 루프의 정체"에서 다룬 "모델이 도구 호출 JSON을 생성하고 런타임이 파싱해서 실행한다"는 루프를 n8n이라는 노코드 툴 위에서 그대로 구현한 것이다.</p>

### Code Tool 정의


<pre><code>return query.numbers.reduce((a, b) => a + b, 0);
</code></pre>

<ul>
  <li><code>specifyInputSchema: true</code>로 입력 스키마를 직접 지정 (<code>{"numbers": [1,2,3,4]}</code>).</li>
  <li><code>description</code>에 "언제 이 도구를 써야 하는지"를 적어두면, 에이전트가 사용자의 발화("더해줘", "합계 알려줘" 등)와 도구를 매칭할 때 이 설명을 활용한다. 도구 description의 품질이 곧 에이전트의 도구 선택 정확도에 직결된다.</li>
</ul>

### 관찰: 로컬 gemma3:e4b의 덧셈 정확도


<p>로컬에서 Ollama로 띄운 <code>gemma3:e4b</code> 모델은 (도구 호출 없이 모델 자체 추론만으로도) 단순 덧셈을 비교적 잘 틀리지 않는 편이었다. 다만 이 워크플로처럼 Tool Calling 구조를 갖춰두면, 모델의 산술 정확도에 의존하지 않고 실제 계산은 결정적인 코드(Code Tool)가 수행하므로 더 안정적이다 — 모델은 "숫자를 추출해서 도구에 넘기는 역할"만 하면 된다.</p>

### 배운 점


<ul>
  <li>n8n의 LangChain 노드는 일반 LangChain 코드의 <code>AgentExecutor</code> + <code>Tool</code> 구성을 그대로 노드 그래프로 옮긴 것에 가깝다. <code>ai_languageModel</code>, <code>ai_tool</code> 같은 연결 타입이 LangChain의 "어떤 컴포넌트가 에이전트에 주입되는가"를 표현한다.</li>
  <li>산술처럼 모델이 가끔 틀릴 수 있는 연산은, 모델에게 직접 계산을 시키기보다 Tool로 위임하는 패턴이 더 안전하다. 모델은 자연어 → 구조화된 입력 변환만 책임진다.</li>
</ul>

### 이 에이전트를 직접 평가해보기


<p>"되긴 되는데 얼마나 잘 되는지"를 숫자로 말하기 위해, 이 덧셈 에이전트로 평가 지표를 직접 계산해봤다.</p>

#### 출력만 보면 안 되는 이유


<p>최종 답이 맞아도, 모델이 도구를 쓰지 않고 머릿속으로 계산해서 우연히 맞혔을 수 있다. 숫자가 복잡해지면 이 우연은 깨진다. "답이 맞았는가"와 "올바른 방법(도구 호출)으로 도달했는가"를 따로 측정해야 한다.</p>

<ul>
  <li><strong>Task Success Rate</strong> — 최종 답이 맞았는지만 보는 가장 거친 지표. <code>성공 케이스 수 / 전체 케이스 수</code>.</li>
  <li><strong>Tool Call Accuracy</strong> — (1) 올바른 도구를 선택했는지, (2) 그 도구에 올바른 인자를 넘겼는지를 측정. 블랙박스로 안 보고 내부 동작을 채점한다.</li>
</ul>

#### 실습 결과: 두 지표가 실제로 갈린 사례


<p>6개 케이스(범위 내 덧셈, 음수 포함 덧셈, 범위 밖 곱하기/나누기, 부분 도구 사용 복합 케이스)를 직접 만들어 돌렸다.</p>

<table>
  <tr><th>케이스</th><th>기대 도구 호출</th><th>실제 도구 호출</th><th>일치?</th></tr>
  <tr><td>1+2+7, 5~10 합</td><td>호출 (2번)</td><td>호출 2번</td><td>✅</td></tr>
  <tr><td>1-2+3</td><td>호출</td><td>호출 안 함</td><td>❌</td></tr>
  <tr><td>2×5</td><td>호출 안 함</td><td>호출 안 함</td><td>✅</td></tr>

  <li><strong>Task Success Rate = 6/6 = 100%</strong> (모든 답이 맞음)</li>
  <li><strong>Tool Call Accuracy = 5/6 ≈ 83%</strong> ("1-2+3"에서 도구 없이 직접 계산함)</li>
</ul>

<p>Task Success만 봤다면 완벽한 에이전트처럼 보였겠지만, 실제로는 음수가 섞인 간단한 연산에서 도구를 안 쓰고 있었다. 위 "로컬 gemma3:e4b의 덧셈 정확도" 관찰과 같은 맥락 — 모델이 가끔은 도구 없이도 맞히지만, 숫자가 복잡해지면 이 우연이 깨질 위험을 Tool Call Accuracy가 미리 잡아준다.</p>

#### 수정: 뺄셈을 음수 덧셈으로 치환하도록 system message에 조건 추가


<p>"1-2+3"에서 도구를 안 쓰던 문제를 system message에 "뺄셈은 음수를 더하는 것으로 치환해서 Code Tool에 넘겨라"는 조건을 추가해서 고쳤다. 예: <code>1-2+3</code> → <code>add(1, -2, 3)</code>으로 변환하라고 명시. 이 조건을 추가한 뒤로는 같은 케이스에서 에이전트가 올바르게 도구를 호출했다.</p>

<p>이건 Tool Call Accuracy가 낮게 나온 원인이 모델의 산술 능력 문제가 아니라 <strong>system message(도구 사용 지침)의 누락</strong>이었다는 뜻이다 — 지표가 "어디를 고쳐야 하는지"를 정확히 가리켜준 사례.</p>

## pytest로 n8n 에이전트 워크플로 자동 테스트하기


### 무엇을 했나


<p>위 "n8n으로 만든 LangChain Agent + Tool 워크플로"에서 만든 덧셈 에이전트를 수동으로 채팅창에 입력하며 확인하는 대신, <code>pytest</code>로 여러 케이스를 한 번에 자동 검증하도록 만들었다.</p>

<pre><code>import pytest
import requests

N8N_CHAT_URL = 'http://localhost:5678/webhook/.../chat'

test_cases = [
    {'input': '1 + 2 + 3', 'expected': 6},
    {'input': '1 부터 100까지 더해줘', 'expected': '5050'},
    {'input': '1 - 2 + 3', 'expected': '2'},
    {'input': '1 부터 100까지 더한 값에 5000을 빼줘', 'expected': '50'}
]

@pytest.mark.parametrize('case', test_cases)
def test_task_success(case):
    resp = requests.post(N8N_CHAT_URL, json={
        'chatInput': case['input'],
        'sessionId': 'automated-test',
    })
    output = resp.json()['output']
    assert str(case['expected']) in output
</code></pre>

### 구조


<ul>
  <li>n8n의 Chat Trigger 노드가 만든 웹훅 URL(<code>/webhook/&lt;id&gt;/chat</code>)에 <code>chatInput</code>과 <code>sessionId</code>를 담아 POST 요청을 보낸다. n8n 채팅 UI가 내부적으로 호출하는 것과 동일한 API를 코드로 직접 호출하는 것이다.</li>
  <li><code>pytest.mark.parametrize</code>로 테스트 케이스 목록을 순회하며, 케이스마다 별도의 테스트로 리포트된다 — 하나가 실패해도 나머지는 계속 실행/리포트된다.</li>
  <li>검증은 응답 JSON의 <code>output</code> 필드에 기대값이 <strong>문자열로 포함되는지</strong>만 본다 (<code>str(case['expected']) in output</code>). 에이전트는 자연어로 답하므로 ("정답은 5050입니다" 등) 정확한 문자열 일치 대신 부분 포함 검사를 쓴 것.</li>
</ul>

### 이게 Task Success Rate를 코드로 만든 것


<p>위 "n8n으로 만든 LangChain Agent + Tool 워크플로"에서 수동으로 표를 만들어 계산했던 <strong>Task Success Rate</strong>(최종 답이 맞는지만 보는 지표)를, pytest 케이스 통과/실패 수로 자동 계산할 수 있게 된 셈이다. <code>pytest</code>를 돌리면 "N개 중 M개 통과"가 바로 나오므로, 워크플로를 수정할 때마다 회귀가 생겼는지 빠르게 확인할 수 있다.</p>

<p>다만 이 테스트는 <strong>Tool Call Accuracy</strong>(올바른 도구를 올바른 인자로 호출했는지)는 측정하지 못한다 — 응답 텍스트만 보고 판단하기 때문에, 모델이 도구 없이 우연히 맞혔는지는 알 수 없다. 그 부분은 n8n 실행 로그를 별도로 까봐야 한다.</p>

### 배운 점


<ul>
  <li>노코드 도구(n8n)로 만든 워크플로도 결국 HTTP API이므로, 일반적인 API 테스트 도구(pytest + requests)로 그대로 자동화할 수 있다.</li>
  <li>자연어 응답을 검증할 때는 정확한 매칭보다 "기대 키워드가 포함되는가" 같은 느슨한 검증이 실용적이다. 단, 너무 느슨하면 거짓 양성(예: "50"이 "5050"에도 포함됨)이 생길 수 있어 기대값 설계에 주의가 필요하다.</li>
</ul>

## Tool Calling 루프의 정체


<p>모델이 "글을 쓰는 것"에서 "행동을 요청하는 것"으로 넘어가는 순간을 정리했다.</p>

### 핵심 제약: 모델은 코드를 실행할 수 없다


<p>Gemma 4 같은 모델은 다음 토큰을 예측할 뿐이다. "날씨를 알려줘"라는 요청을 받아도 모델이 실제로 API를 호출하지 않는다. 대신 <em>"이 함수를 이렇게 호출해야 한다"</em>는 구조화된 텍스트를 생성할 뿐이고, 이 텍스트를 파싱해서 실제로 함수를 실행하는 책임은 항상 개발자 코드에 있다.</p>

<blockquote>Google 공식 문서: "A Gemma model cannot execute code on its own." (<a href="https://ai.google.dev/gemma/docs/capabilities/text/function-calling-gemma4">ai.google.dev</a>)</blockquote>

### 4단계 루프


<div class="mermaid">
sequenceDiagram
  participant Dev as 개발자 코드
  participant Model as 모델
  Dev->>Model: 1. tool 선언 (이름·설명·파라미터 스키마)
  Dev->>Model: 사용자 메시지
  Model-->>Dev: 2. tool_calls (구조화된 "요청")
  Dev->>Dev: 3. 함수 이름·인자를 꺼내 직접 실행
  Dev->>Model: 4. role: "tool" 메시지로 결과 반환
  Model-->>Dev: 자연어 최종 응답
</div>

<table>
  <tr><th>단계</th><th>내용</th></tr>
  <tr><td>1. tool 선언</td><td>사용 가능한 함수의 이름·설명·파라미터 스키마를 모델에게 알려준다</td></tr>
  <tr><td>2. 모델이 "요청"</td><td>모델은 직접 실행하지 않고 <code>tool_calls</code>라는 구조화된 데이터를 응답에 담아 돌려준다</td></tr>
  <tr><td>3. 직접 실행</td><td><code>tool_calls</code> 안의 함수 이름·인자를 꺼내 내 코드가 직접 호출한다</td></tr>
  <tr><td>4. 결과 반환</td><td>실행 결과를 <code>role: "tool"</code> 메시지로 대화 기록에 추가하고 다시 모델을 호출한다</td></tr>
</table>

<pre><code>from ollama import chat

def get_temperature(city: str) -> str:
    temperatures = {"Seoul": "26°C", "Tokyo": "24°C"}
    return temperatures.get(city, "데이터 없음")

messages = [{"role": "user", "content": "서울 기온이 어때?"}]
response = chat(model="gemma4:e4b", messages=messages, tools=[get_temperature])

if response.message.tool_calls:
    call = response.message.tool_calls[0]
    result = get_temperature(**call.function.arguments)

    messages.append(response.message)
    messages.append({"role": "tool", "tool_name": call.function.name, "content": str(result)})
    final = chat(model="gemma4:e4b", messages=messages, tools=[get_temperature])
    print(final.message.content)  # "현재 서울의 기온은 26°C입니다."
</code></pre>

<p>실제로 <code>gemma4:e4b</code>로 돌려보면 모델이 <code>thinking</code> 필드에 "도시 이름을 추출하고 어떤 tool을 쓸지 정한다"는 추론을 남기고, <code>tool_calls=[ToolCall(function=Function(name='get_temperature', arguments={'city': 'Seoul'}))]</code> 형태로 응답한다. 두 번째 호출에서는 <code>tool_calls=None</code>이 되고 <code>content</code>에 자연어 답이 채워진다.</p>

### LangChain은 같은 루프를 이름만 바꿔서 감싼다


<p>새 개념이 아니라 같은 4단계에 LangChain식 이름표가 붙는 것뿐이다.</p>

<table>
  <tr><th>단계</th><th>raw ollama</th><th>LangChain</th></tr>
  <tr><td>1. tool 선언</td><td><code>tools=[get_temperature]</code></td><td><code>@tool</code> 데코레이터 + <code>.bind_tools([...])</code></td></tr>
  <tr><td>2. 모델의 요청</td><td><code>response.message.tool_calls</code></td><td><code>AIMessage.tool_calls</code></td></tr>
  <tr><td>3. 직접 실행</td><td><code>get_temperature(**call.function.arguments)</code></td><td><code>get_temperature.invoke(tool_call['args'])</code></td></tr>
  <tr><td>4. 결과 반환</td><td><code>{"role": "tool", ...}</code></td><td><code>ToolMessage(tool_call_id=..., content=...)</code></td></tr>
</table>

<pre><code>from langchain.messages import AIMessage, ToolMessage
from langchain.tools import tool
from langchain_ollama import ChatOllama

@tool
def get_temperature(city: str) -> str:
    """주어진 도시의 현재 기온을 반환한다.

    Args:
        city: 영어 도시 이름 (예: "Seoul", "Tokyo")
    """
    temperatures = {"Seoul": "26°C", "Tokyo": "24°C"}
    return temperatures.get(city, "데이터 없음")

llm = ChatOllama(model="gemma4:e4b", temperature=0).bind_tools([get_temperature])
result = llm.invoke("서울 기온이 어때?")

if isinstance(result, AIMessage) and result.tool_calls:
    tool_call = result.tool_calls[0]
    tool_result = get_temperature.invoke(tool_call['args'])

    messages = [result, ToolMessage(tool_call_id=tool_call['id'], content=str(tool_result))]
    final = llm.invoke(messages)
    print(final.content)
</code></pre>

### 헷갈리기 쉬운 지점: <code>.</code> 접근 vs <code>[]</code> 접근


<p>raw ollama의 <code>tool_calls</code>는 Pydantic 객체 리스트라서 <code>call.function.name</code>처럼 점(<code>.</code>)으로 접근한다. 반면 LangChain의 <code>AIMessage.tool_calls</code>는 <strong>dict의 리스트</strong>라서 <code>tool_call['name']</code>처럼 <code>[]</code>로 접근해야 한다. "객체는 <code>.</code>, 딕셔너리는 <code>[]</code>"라는 기본 규칙이 두 라이브러리 사이에서 그대로 적용되는 것뿐이다.</p>

<pre><code># raw ollama
call.function.name        # 객체 속성 접근
call.function.arguments   # {'city': 'Seoul'}

# LangChain
tool_call['name']         # dict 접근
tool_call['args']         # {'city': 'Seoul'}
</code></pre>

<p><code>**args</code>로 딕셔너리를 키워드 인자로 풀어 넘기는 것도 두 경우 모두 동일하게 쓰인다(<code>get_temperature(**args)</code> ≡ <code>get_temperature(city='Seoul')</code>).</p>

### 실전 교훈


<ul>
  <li>모델은 절대 코드를 직접 실행하지 않는다 — <code>tool_calls</code>는 항상 "요청"일 뿐, 실행은 항상 내 코드의 책임이다.</li>
  <li>모델이 프롬프트의 언어를 인자 값에 그대로 반영할 수 있다(예: "서울" vs "Seoul"). tool 함수의 키·enum 값은 한 언어로 고정하고, docstring에 매핑 규칙을 명시해 두는 게 안전하다.</li>
</ul>
