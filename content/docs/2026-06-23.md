## Building Effective Agents

### 에이전트란 무엇인가

> 가장 성공적인 구현들은 일관되게 복잡한 프레임워크나 전문화된 라이브러리를 사용하지 않았습니다. 대신, 이들은 단순하고 조합 가능한 패턴으로 구축하고 있었습니다.

"에이전트"는 여러 가지 방식으로 정의될 수 있다.

- 장시간에 걸쳐 독립적으로 작동하며, 다양한 도구를 사용해 복잡한 작업을 수행하는 완전히 자율적인 시스템
- 미리 정의된 워크플로우를 따르는 보다 규정된 형태의 구현

Anthropic은 이러한 변형들을 모두 **에이전틱 시스템(agentic systems)**으로 분류하지만, 워크플로우와 에이전트 사이에 중요한 아키텍처적 구분을 둔다.

- **워크플로우**: LLM과 도구들이 미리 정의된 코드 경로를 통해 오케스트레이션되는 시스템
- **에이전트**: LLM이 스스로의 프로세스와 도구 사용을 동적으로 지시하며, 작업 수행 방식에 대한 통제권을 유지하는 시스템

### 언제 (그리고 언제 사용하지 말아야) 에이전트를 쓸 것인가

> 에이전틱 시스템은 흔히 더 나은 작업 성능을 위해 지연 시간과 비용을 대가로 치르며, 이러한 트레이드오프가 언제 합리적인지를 고려해야 합니다.

- LLM 애플리케이션을 만들 때는 가능한 가장 단순한 해결책을 찾고, 필요할 때만 복잡성을 높인다.
- 복잡성이 더 필요한 경우, 워크플로우는 명확히 정의된 작업에 예측 가능성·일관성을 주는 반면, 에이전트는 대규모로 유연성과 모델 주도 의사결정이 필요할 때 더 나은 선택이다.
  - 다만 많은 애플리케이션에서는 검색(retrieval)과 인컨텍스트 예시로 단일 LLM 호출을 최적화하는 것만으로도 충분한 경우가 많다.

### Building block: 증강된 LLM (The augmented LLM)

<div class="mermaid">
flowchart LR
    In((In)) --> LLM[LLM]
    LLM --> Out((Out))
    LLM <-.Query/Results.-> Retrieval[Retrieval]
    LLM <-.Call/Response.-> Tools[Tools]
    LLM <-.Read/Write.-> Memory[Memory]
</div>

- 에이전틱 시스템의 기본 구성 요소는 검색(retrieval), 도구(tools), 메모리(memory) 같은 증강 기능으로 강화된 LLM이다.
- 증강 기능을 구현하는 방법은 다양한데, 그중 하나가 최근 공개된 Model Context Protocol(MCP)을 활용하는 것이다.
- 구현 시 핵심은 두 가지다: 이러한 기능들을 특정 사용 사례에 맞게 조정하는 것, 그리고 LLM이 쉽고 잘 문서화된 인터페이스를 제공받도록 보장하는 것.

### Workflow: Prompt chaining

<div class="mermaid">
flowchart LR
    In((In)) --> C1[LLM Call 1]
    C1 -->|Output 1| Gate{Gate}
    Gate -->|Pass| C2[LLM Call 2]
    Gate -.->|Fail| Exit((Exit))
    C2 -->|Output 2| C3[LLM Call 3]
    C3 --> Out((Out))
</div>

작업을 일련의 단계로 분해하며, 각 LLM 호출은 이전 호출의 결과물을 처리한다. 중간 단계에 프로그래밍 방식의 검증("gate")을 추가해 프로세스가 올바른 방향으로 가고 있는지 확인할 수 있다.

- 주된 목표는 각 LLM 호출을 더 쉬운 작업으로 만들어서, 지연 시간을 대가로 더 높은 정확도를 얻는 것이다.
- **사용 시점**: 작업이 고정된 하위 작업들로 쉽고 명확하게 분해될 수 있는 상황에 이상적이다.

### Workflow: Routing

<div class="mermaid">
flowchart LR
    In((In)) --> Router[LLM Call Router]
    Router --> C1[LLM Call 1] --> Out((Out))
    Router -.-> C2[LLM Call 2] -.-> Out
    Router -.-> C3[LLM Call 3] -.-> Out
</div>

입력을 분류하고 그에 특화된 후속 작업으로 전달한다. 관심사의 분리(separation of concerns)를 가능하게 하며, 더 전문화된 프롬프트를 구축할 수 있게 해준다.

- 이 워크플로우가 없으면, 한 가지 유형의 입력에 대한 최적화가 다른 입력들의 성능을 저해할 수 있다.
- **사용 시점**: 명확히 구분되어 별도로 처리하는 것이 더 나은 복잡한 작업이고, 그 분류가 (LLM이든 전통적인 분류 모델/알고리즘이든) 정확하게 처리될 수 있는 경우다.

### Workflow: Parallelization

<div class="mermaid">
flowchart LR
    In((In)) --> C1[LLM Call 1] --> Agg[Aggregator]
    In --> C2[LLM Call 2] --> Agg
    In --> C3[LLM Call 3] --> Agg
    Agg --> Out((Out))
</div>

LLM들이 동시에 하나의 작업을 수행하고, 그 결과물을 프로그래밍 방식으로 종합한다. 두 가지 주요 형태가 있다.

- **분할(Sectioning)**: 작업을 독립적인 하위 작업들로 나누어 병렬로 실행하는 방식이다.
- **투표(Voting)**: 동일한 작업을 여러 번 실행하여 다양한 결과물을 얻는 방식이다.

**사용 시점**: 분할된 하위 작업들을 속도를 위해 병렬로 처리할 수 있을 때, 또는 더 높은 신뢰도의 결과를 얻기 위해 여러 관점/시도가 필요할 때다.

> (내 생각) 약간 Routing과도 연결되는 생각인 것 같다. + 라우팅이 가장 작은 단위로 오케스트레이터-워커, 병렬화 등에 적용되는 방법인듯..?

### Workflow: Orchestrator-workers

<div class="mermaid">
flowchart LR
    In((In)) --> Orch[Orchestrator]
    Orch -.-> C1[LLM Call 1] -.-> Syn[Synthesizer]
    Orch -.-> C2[LLM Call 2] -.-> Syn
    Orch -.-> C3[LLM Call 3] -.-> Syn
    Syn --> Out((Out))
</div>

중앙 LLM(오케스트레이터)이 작업을 동적으로 분해하고, 이를 워커 LLM들에게 위임한 후 그 결과들을 종합한다.

- **사용 시점**: 필요한 하위 작업들을 예측할 수 없는 복잡한 작업 (예: 코딩 작업에서 변경이 필요한 파일 수와 각 파일의 변경 내용은 작업마다 다름).
- 구조적으로는 병렬화와 유사하지만, 핵심 차이는 유연성에 있다. 하위 작업이 미리 정의되어 있지 않고, 오케스트레이터가 입력을 보고 그때그때 결정한다.

### Workflow: Evaluator-optimizer

<div class="mermaid">
flowchart LR
    In((In)) --> Gen[LLM Call Generator]
    Gen -->|Solution| Eval[LLM Call Evaluator]
    Eval -->|Rejected + Feedback| Gen
    Eval -->|Accepted| Out((Out))
</div>

하나의 LLM 호출이 응답을 생성하고, 다른 LLM 호출이 반복적인 루프 안에서 평가와 피드백을 제공한다.

- **사용 시점**: 명확한 평가 기준이 있고, 반복적인 개선이 측정 가능한 가치를 줄 때. 적합성을 나타내는 두 신호는 (1) 사람이 피드백을 명확히 표현했을 때 LLM 응답이 분명히 개선될 수 있다는 것, (2) LLM이 그 피드백을 직접 제공할 수 있다는 것이다.

### Agents

<div class="mermaid">
flowchart LR
    Human((Human)) <-.-> LLMCall[LLM Call]
    LLMCall -->|Action| Environment((Environment))
    Environment -->|Feedback| LLMCall
    LLMCall --> Stop[Stop]
</div>

LLM이 복잡한 입력을 이해하고, 추론·계획을 수행하며, 도구를 신뢰성 있게 사용하고, 오류로부터 회복하는 핵심 역량들이 성숙해지면서 production 환경에 에이전트가 등장하고 있다.

- 에이전트는 인간 사용자의 명령이나 인간과의 상호작용적 논의로부터 작업을 시작한다.
- 실행 중에는 각 단계마다 환경으로부터 "그라운드 트루스"(도구 호출 결과, 코드 실행 결과 등)를 얻어 자신의 진행 상황을 평가하는 것이 매우 중요하다.
- 본질적으로는 환경 피드백을 바탕으로 루프 안에서 도구를 사용하는 LLM일 뿐이다. 따라서 도구 모음(toolsets)과 그 문서를 명확하고 신중하게 설계하는 것이 매우 중요하다.
- **사용 시점**: 필요한 단계 수를 예측하기 어렵거나 불가능한 개방형 문제, 고정된 경로를 하드코딩할 수 없는 경우.
- 자율적인 특성은 더 높은 비용과 오류 누적 가능성을 의미한다.

코딩 에이전트를 예로 들면, 실제 실행 루프는 사람·인터페이스·LLM·환경 사이의 다음과 같은 시퀀스로 펼쳐진다.

<div class="mermaid">
sequenceDiagram
    participant Human
    participant Interface
    participant LLM
    participant Environment
    Human->>Interface: Query
    loop Until tasks clear
        Human->>Interface: Clarify
        Interface->>Human: Refine
    end
    Interface->>LLM: Send context
    LLM->>Environment: Search files
    Environment->>LLM: Return paths
    loop Until tests pass
        LLM->>Environment: Write code
        Environment->>LLM: Status
        LLM->>Environment: Test
        Environment->>LLM: Results
    end
    LLM->>Interface: Complete
    Interface->>Human: Display
</div>

### 패턴 조합 및 커스터마이징

> 다시 한번 강조하자면: 복잡성을 추가하는 것은 그것이 결과를 분명히 개선시킨다는 것이 입증될 때에만 고려해야 합니다.

이 구성 요소들은 규정처럼 따라야 하는 것이 아니라, 개발자가 사용 사례에 맞게 변형·결합할 수 있는 공통 패턴이다. 성공의 핵심은 다른 LLM 기능과 마찬가지로 성능을 측정하고 구현을 반복적으로 개선해 나가는 것.

### Summary

> LLM 분야에서의 성공은 가장 정교한 시스템을 구축하는 것이 아닙니다. 여러분의 필요에 맞는 올바른 시스템을 구축하는 것입니다.

단순한 프롬프트로 시작해 포괄적인 평가로 최적화하고, 더 단순한 해결책이 부족할 때에만 다단계 에이전틱 시스템을 추가한다.

에이전트를 구현할 때 따라야 할 세 가지 핵심 원칙:

1. 에이전트 설계에서 단순함을 유지하라.
2. 에이전트의 계획 단계를 명시적으로 드러냄으로써 투명성을 우선시하라. (에이전트가 작업을 어떻게 처리하고 있는지 그 과정을 사용자/개발자가 볼 수 있게 해야 한다는 의미)
3. 철저한 도구 문서화와 테스트를 통해 에이전트-컴퓨터 인터페이스(ACI)를 신중하게 설계하라.

프레임워크는 빠르게 시작하는 데 도움이 될 수 있지만, production 단계로 나아갈 때에는 추상화 계층을 줄이고 기본적인 구성 요소들로 직접 구축하는 것을 주저하지 않는다. 이 원칙들을 따르면 강력할 뿐만 아니라 신뢰할 수 있고, 유지보수가 가능하며, 사용자의 신뢰를 받는 에이전트를 만들 수 있다.

<summary>Appendix: 도구를 위한 프롬프트 엔지니어링</summary>

> 도구의 정의와 명세는 전체 프롬프트만큼이나 프롬프트 엔지니어링에 신경을 써야 합니다.

도구 형식을 결정할 때 제안되는 사항:

- 모델이 스스로를 막다른 곳으로 몰아넣기 전에 "생각"할 수 있는 충분한 토큰을 제공하라.
- 형식을 모델이 인터넷상의 텍스트에서 자연스럽게 접해온 것과 가깝게 유지하라.
- 수천 줄의 코드를 정확히 세야 하거나, 작성하는 코드에 문자열 이스케이핑을 해야 하는 것과 같은 형식상의 "오버헤드"가 없도록 하라.

도구 정의를 개선하는 방법:

- 모델의 입장에서 생각해 보라. 설명과 파라미터만 보고 이 도구를 어떻게 사용해야 하는지가 명확한가, 아니면 신중하게 생각해야 하는가? 후자라면 모델에게도 마찬가지일 가능성이 크다. 좋은 도구 정의는 흔히 사용 예시, 엣지 케이스, 입력 형식 요구사항, 다른 도구들과의 명확한 경계를 포함한다.
- 파라미터 이름이나 설명을 어떻게 바꾸면 더 명확해질 수 있을까? 팀의 신입 개발자를 위해 훌륭한 docstring을 작성하는 것이라고 생각하라. 비슷한 도구를 여러 개 사용할 때 특히 중요하다.
- 모델이 도구를 어떻게 사용하는지 테스트하라: Anthropic의 워크벤치(workbench)에서 많은 예시 입력을 실행해 모델이 어떤 실수를 하는지 확인하고, 반복적으로 개선하라.
- 도구에 포카요케(poka-yoke, 실수 방지) 원칙을 적용하라. 실수를 하기 어렵게 인자(arguments)를 설계하라.

## LangChain + Neo4j로 지식 그래프 만들기 (Graph RAG 기초)

Graph RAG는 벡터 검색 대신(또는 함께) 그래프 데이터베이스에 엔티티와 관계를 저장해두고, 그래프 탐색(traversal)으로 연관 정보를 가져오는 방식이다. `langchain_neo4j`의 `Neo4jGraph`를 쓰면 Cypher 쿼리를 LangChain 코드에서 바로 실행할 수 있다.

### 그래프 연결

```python
from langchain_neo4j import Neo4jGraph

graph = Neo4jGraph(
    url="neo4j://localhost:7687",
    username="neo4j",
    password="password123",
)
```

`graph.query(cypher, params=...)` 형태로 Cypher를 실행하며, 결과는 dict 리스트로 반환된다.

### 노드 생성: MERGE로 멱등성 확보

`CREATE` 대신 `MERGE`를 쓰면 같은 이름의 노드가 이미 있을 때 중복 생성하지 않고 기존 노드를 재사용한다. 같은 스크립트를 여러 번 실행해도 안전하다.

```python
graph.query("""
MERGE (p:Person:HistoricalFigure {name: $name})
SET p.born = $born, p.died = $died, p.role = $role
""", params={"name": "세종대왕", "born": 1397, "died": 1450, "role": "왕"})
```

- 한 노드에 `Person`, `HistoricalFigure`처럼 레이블을 여러 개 붙일 수 있다(다중 분류).
- `MERGE`는 매칭 키(`{name: $name}`)만으로 노드를 찾고, 나머지 속성은 `SET`으로 별도 설정하는 패턴이 일반적이다. `MERGE` 절 안에 가변적인 속성(예: `died`가 없는 인물)까지 넣으면 매칭 조건이 달라져 같은 인물이 중복 생성될 수 있기 때문이다.

### 관계 생성: 동적 관계 타입

Cypher의 관계 타입은 파라미터 바인딩이 안 되므로, 관계 타입 자체를 동적으로 넣어야 할 때는 f-string으로 쿼리 문자열을 만든다.

```python
relationships = [
    ("세종대왕", "CREATED", "훈민정음"),
    ("세종대왕", "ESTABLISHED", "집현전"),
]

for source, rel_type, target in relationships:
    graph.query(f"""
        MATCH (a {{name: $source}})
        MATCH (b {{name: $target}})
        MERGE (a)-[:{rel_type}]->(b)
    """, params={"source": source, "target": target})
```

> 관계 타입은 노드 레이블과 마찬가지로 Cypher 문법 구조의 일부라 파라미터화할 수 없다. 다만 이 f-string 패턴은 `rel_type`이 외부 입력(사용자 입력 등)에서 직접 올 경우 Cypher 인젝션 위험이 있으므로, 실제 서비스에서는 허용된 관계 타입 목록으로 화이트리스트 검증을 거쳐야 한다.

### 그래프 탐색 쿼리

구축한 그래프는 일반적인 패턴 매칭으로 조회한다.

```cypher
MATCH (sejong:Person {name: "세종대왕"})-[r]->(related)
RETURN type(r) AS 관계, labels(related) AS 유형, related.name AS 대상
```

- `-[r]->`처럼 관계 타입을 비워두면 모든 관계를 한 번에 가져올 수 있다.
- 2단계(multi-hop) 관계 탐색도 한 줄의 패턴으로 표현된다. 아래는 "세종대왕이 설립한 기관에서 일한 학자"를 찾는 쿼리다.

```cypher
MATCH (sejong:Person {name: "세종대왕"})-[:ESTABLISHED]->(org:Organization)<-[:WORKED_AT]-(scholar:Person)
RETURN org.name AS 기관, scholar.name AS 학자
```

이런 멀티홉 패턴 매칭이 벡터 검색만으로는 표현하기 어려운, Graph RAG의 핵심 장점이다. "A가 설립한 기관에서 일한 사람"처럼 관계를 명시적으로 연결해 가는 질의는 그래프 구조에서 자연스럽게 풀리지만, 임베딩 유사도 검색만으로는 이런 다단계 관계 추론을 안정적으로 재현하기 어렵다.
