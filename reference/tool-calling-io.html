Tool Calling I/O 구조 빠른 참조

4단계 루프

1. tools=[...]       → 모델에게 사용 가능한 함수 알려주기
2. response.message.tool_calls → 모델이 "호출해달라"고 요청 (실행 아님)
3. 직접 함수 실행()   → 너의 코드가 실제로 호출
4. role:"tool" 메시지 추가 후 재호출 → 모델이 결과로 자연어 답 생성

응답 객체 구조 (Ollama Python SDK)

response.message
  .content        # 자연어 답 (tool 호출 시 보통 빈 문자열)
  .thinking        # 모델의 내부 추론 (thinking 지원 모델만)
  .tool_calls      # [ToolCall(...), ...] 리스트
      [0].function
          .name        # 'get_temperature'
          .arguments   # {'city': 'Seoul'}  ← dict

속성 접근(.) vs 딕셔너리 접근([])

대상접근 방식예시
객체(클래스 인스턴스).속성명call.function.name
메서드(동작).메서드명()s.upper()
딕셔너리[키]arguments["city"]

Ollama/LangChain 응답은 보통 dict가 아니라 객체(Pydantic 모델)라서 점(.) 표기를 쓴다.

딕셔너리 언패킹 **

args = {'city': 'Seoul'}
get_temperature(**args)
# 위와 동일:
get_temperature(city='Seoul')

**딕셔너리는 키-값 쌍을 모두 키워드 인자로 펼친다. tool이 인자를 몇 개 받든 한 줄로 그대로 넘길 수 있다.

참고: *리스트는 위치 인자로 펼치는 짝꿍 문법.

raw ollama vs LangChain 비교

단계raw ollamaLangChain (ChatOllama)
tool 선언tools=[fn]@tool + .bind_tools([fn])
모델 요청response.message.tool_calls (객체 리스트)AIMessage.tool_calls (dict 리스트)
인자 꺼내기call.function.arguments (.)tool_call['args'] ([])
결과 반환{"role": "tool", ...}ToolMessage(tool_call_id=..., content=...)

주의: LangChain의 tool_call은 dict라서 .name이 아니라 ['name']으로 접근한다.

실전 교훈

  • 모델은 절대 코드를 직접 실행하지 않는다 — tool_calls는 항상 "요청"일 뿐.
  • 모델이 프롬프트 언어를 인자 값에 그대로 반영할 수 있다 (예: "서울" vs "Seoul"). tool 함수의 키/enum은 한 언어로 고정하고 docstring에 매핑을 명시하는 게 안전.
