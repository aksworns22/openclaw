
## Introduction

> Good **evaluations** help teams ship AI agents more confidently.

평가가 없으면 반응적인 루프에 갇히기 쉽다—프로덕션 환경에서만 문제를 발견하게 되고, 거기서는 하나의 실패를 고치면 또 다른 실패가 생겨난다.

## The structure of an evaluation

![[Pasted image 20260619223942.png]]

에이전트는 여러 턴에 걸쳐 도구를 사용하면서 환경의 상태를 변경하고 그 과정에서 적응해 나간다—이는 실수가 전파되고 누적될 수 있다는 의미다.

최첨단 모델은 또한 정적인 평가의 한계를 뛰어넘는 창의적인 해법을 찾아낼 수도 있다.

예를 들어, Opus 4.5는 항공편 예약에 관한 𝜏2-bench 문제를 정책상의 빈틈을 발견함으로써 해결했다.

이는 작성된 평가 기준상으로는 "실패"였지만, 실제로는 사용자에게 더 나은 해결책을 제시한 것이었다.

## Definitions for building agent evaluation :

![[Pasted image 20260619224342.png]]

- task는 정의된 입력값과 성공 기준을 가진 하나의 단일 테스트다
- 과제에 대한 각 시도를 트라이얼(trial)이라 한다.
- 채점자(grader)는 에이전트 성능의 특정 측면을 점수화하는 로직이다.
- transcript(trace, trajectory)는 출력, 도구 호출, 추론, 중간 결과, 그리고 그 외 모든 상호작용을 포함한 트라이얼의 전체 기록이다.
- 결과(outcome)는 트라이얼이 끝났을 때 환경의 최종 상태다.
- 평가 하니스(evaluation harness)는 평가를 처음부터 끝까지 실행하는 인프라
- agent harness(scaffold)는 모델이 에이전트로서 행동할 수 있게 해주는 시스템이다
- 평가 스위트(evaluation suite)는 특정 능력이나 행동을 측정하기 위해 설계된 과제들의 모음

## Why build evaluations?

> 비용은 처음부터 눈에 보이지만 이점은 나중에야 누적되기 때문에, 평가가 가진 복리적 가치는 놓치기 쉽다.

- 더 엄격한 평가는 오히려 출시 속도를 늦추는 오버헤드처럼 보일 수도 있다.
- 평가가 없을 때 한계점은 흔히 사용자가 변경 후 에이전트가 더 나빠진 것 같다고 보고할 때 찾아오는데, 이때 팀은 추측과 시행착오 외에는 검증할 방법이 없는 "맹목 비행" 상태에 놓인다.
- 팀은 실제 퇴보와 잡음을 구분할 수도 없고, 출시 전에 수백 가지 시나리오에 대해 변경 사항을 자동으로 테스트할 수도 없으며, 개선 정도를 측정할 수도 없다.
- 일부 팀은 개발 초기에 평가를 만들고, 다른 팀은 평가가 에이전트 개선의 병목이 될 정도로 규모가 커졌을 때 한 번에 추가한다.
	- 평가는 기대되는 행동을 명시적으로 코드화한다는 점에서 에이전트 개발 초기에 특히 유용하다.
- 더 강력한 모델이 출시되면, 평가가 없는 팀은 몇 주에 걸친 테스트에 직면하지만, 평가를 갖춘 경쟁사는 모델의 강점을 빠르게 파악하고 프롬프트를 조정하여 며칠 만에 업그레이드할 수 있다.
- 평가가 일단 존재하면, 기준선과 회귀 테스트를 별도 비용 없이 얻게 된다: 지연 시간, 토큰 사용량, 과제당 비용, 오류율 등을 고정된 과제 모음을 통해 추적할 수 있다.

## Types of graders for agents

> 효과적인 평가 설계의 핵심 요소는 그 작업에 맞는 적절한 채점자를 선택하는 것

- 에이전트 평가는 일반적으로 세 가지 유형의 채점자를 결합한다
	- 코드 기반
	- 모델 기반
	- 사람
- 각 채점자는 트랜스크립트나 결과 중 일부를 평가

- 역량 평가(또는 "품질" 평가)는 "이 에이전트가 무엇을 잘할 수 있는가?"를 묻는다.
- 회귀 평가는 "에이전트가 예전에 처리했던 모든 과제를 여전히 처리할 수 있는가?"를 묻고, 거의 100%의 통과율을 가져야 한다.
	- 에이전트가 출시되고 최적화된 후에는, 통과율이 높아진 역량 평가가 "졸업"하여 지속적으로 실행되며 변화(drift)를 감지하는 회귀 스위트가 될 수 있다.
	- 한때 "이걸 도대체 할 수 있는가?"를 측정했던 과제들이 이제는 "이걸 여전히 안정적으로 할 수 있는가?"를 측정하게 된다.

### Evaluating coding agents

> 코딩 에이전트는 마치 인간 개발자처럼 코드베이스를 탐색하고 명령을 실행하며 코드를 작성, 테스트, 디버깅한다.

- 현대 코딩 에이전트를 위한 효과적인 평가는 보통 명확하게 명세된 과제, 안정적인 테스트 환경, 그리고 생성된 코드에 대한 철저한 테스트에 의존한다.
- 결정론적 채점자는 코딩 에이전트에 자연스럽게 어울리는데, 소프트웨어는 일반적으로 평가하기가 단순하기 때문이다: 코드가 실행되는가, 테스트를 통과하는가?
- 아래에 예시로 제시된 YAML 파일에서 보듯이, 이 에이전트는 채점자(grader)와 지표(metric)를 모두 활용해 평가할 수 있다.
	- 실제로는 코딩 평가가 보통 정확성 검증을 위한 단위 테스트와 전반적인 코드 품질 평가를 위한 LLM 루브릭에 의존하며, 추가적인 채점자와 지표는 필요한 경우에만 더해진다.


### Evaluating conversational agents

- 대화형 에이전트를 위한 효과적인 평가는 보통 검증 가능한 최종 상태 결과와, 과제 완수 및 상호작용 품질을 모두 포착하는 루브릭에 의존한다.
	- 다른 대부분의 평가와 달리, 이는 흔히 사용자를 시뮬레이션하는 두 번째 LLM을 필요로 한다.
	- 우리는 이러한 접근법을 정합성 감사(alignment auditing) 에이전트에 활용하여, 길고 적대적인 대화를 통해 모델을 스트레스 테스트한다.
- 대화형 에이전트의 성공은 다차원적일 수 있다: 티켓이 해결되었는가(상태 확인), 10턴 이내에 끝났는가(트랜스크립트 제약), 그리고 어조가 적절했는가(LLM 루브릭)?
