> Context is a critical but finite resource for AI agents.

원문: https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents

## Slack 학습 정리 - 2026-07-01

### 현재까지의 학습 범위

- 주제: Effective context engineering for AI agents
- 핵심 문제: 컨텍스트는 AI 에이전트 성능에 중요한 자원이지만 용량 제한이 있다.
- 학습 방향: 에이전트가 안정적으로 작동하도록 필요한 컨텍스트를 선별하고 관리하는 전략을 정리한다.

### 컨텍스트의 정의

여기서 컨텍스트란 LLM이 응답을 생성할 때 입력으로 제공되는 토큰들의 집합이다.

컨텍스트 엔지니어링의 과제는 LLM이 가진 제약 속에서 이 토큰들을 최대한 효율적으로 활용해, 원하는 결과를 일관되게 얻을 수 있도록 최적화하는 것이다.

즉, 특정 시점에 LLM이 볼 수 있는 전체 상태를 고려하고, 그 상태가 어떤 행동을 이끌어낼 수 있을지까지 함께 생각하는 일이다.

### Prompt engineering vs. context engineering

프롬프트 엔지니어링은 LLM이 최상의 결과를 낼 수 있도록 지시문(prompt)을 작성하고 구성하는 방법이다.

반면 컨텍스트 엔지니어링은 LLM이 추론하는 동안 최적의 토큰, 즉 정보 집합을 선별하고 유지하는 전략이다. 여기에는 프롬프트뿐 아니라 시스템 지시문, 도구, MCP(Model Context Protocol), 외부 데이터, 대화 기록처럼 컨텍스트에 포함될 수 있는 모든 정보가 들어간다.

여러 차례의 추론을 수행하고 장기간 동작하는 AI 에이전트를 만들수록, 전체 컨텍스트 상태를 관리하는 전략이 중요해진다. 반복적으로 실행되는 에이전트는 다음 추론 단계에서 관련될 수 있는 정보를 계속 생성하며, 이 정보는 매 단계에서 다시 정제되어야 한다.

한 줄로 정리하면, 프롬프트 엔지니어링은 "어떻게 말할 것인가"에 가깝고, 컨텍스트 엔지니어링은 "무엇을 보게 할 것인가"에 가깝다.

### 관련 이미지 메모

Slack 스레드에 추가된 비교 이미지는 prompt engineering과 context engineering for agents의 차이를 시각적으로 보여준다.

- 왼쪽의 prompt engineering은 단일 요청에서 context window 안에 system prompt와 user message를 넣고 바로 응답을 얻는 구조에 가깝다.
- 오른쪽의 context engineering for agents는 문서, 도구, 메모리 파일, 도메인 지식, 지시문, 대화 기록 등 여러 후보 컨텍스트 중 현재 단계에 필요한 것만 선별해 context window에 넣는 구조다.
- 에이전트는 응답만 하는 것이 아니라 tool call을 수행하고, 그 결과가 다시 다음 컨텍스트 후보로 들어간다.

따라서 컨텍스트 엔지니어링은 한 번 잘 넣고 끝나는 작업이 아니라, 실행 결과와 대화 흐름에 따라 정보 흐름을 계속 선별, 유지, 갱신하는 작업이다.

### 왜 capable agent를 만들 때 중요한가

`needle-in-a-haystack` 유형의 벤치마크 연구에서는 `context rot`이라는 현상이 발견되었다. 이는 컨텍스트 윈도우에 포함된 토큰이 많아질수록, 모델이 그 안의 정보를 정확히 찾아내고 기억하는 능력이 떨어지는 현상이다.

따라서 컨텍스트는 단순히 많이 넣으면 좋은 저장 공간이 아니라, 한정되어 있고 많이 사용할수록 추가 효과가 점점 줄어드는 자원으로 다루어야 한다.

사람의 작업 기억(working memory)에 용량 제한이 있는 것처럼, LLM에도 많은 컨텍스트를 처리할 때 사용하는 `attention budget`이 있다.

LLM은 Transformer 구조를 기반으로 하며, 이 구조에서는 모든 토큰이 컨텍스트 안의 다른 모든 토큰을 참고(attend)할 수 있다. 컨텍스트가 길어질수록 모델은 더 많은 토큰 간 관계를 고려해야 하므로, 각 관계에 집중할 수 있는 능력이 분산된다.

결국 컨텍스트의 크기와 집중력 사이에는 자연스러운 트레이드오프가 있다. 긴 컨텍스트는 더 많은 정보를 담을 수 있지만, 동시에 중요한 정보를 놓치거나 관계를 흐리게 만들 위험도 커진다.

또한 모델은 학습 과정에서 긴 시퀀스보다 짧은 시퀀스를 훨씬 많이 접하며, 이를 바탕으로 어텐션 패턴을 형성한다. 그래서 긴 컨텍스트를 제공할 수 있다는 것과 긴 컨텍스트를 항상 안정적으로 잘 활용한다는 것은 별개의 문제다.

### The anatomy of effective context

LLM은 사용할 수 있는 `attention budget`이 제한되어 있다. 따라서 좋은 컨텍스트 엔지니어링이란 원하는 결과를 얻을 가능성을 최대화하면서도, 가장 적은 수의 정보 밀도 높은(high-signal) 토큰을 찾아내는 일이다.

시스템 프롬프트는 매우 명확해야 하며, 에이전트에게 적절한 수준(right altitude)에서 정보를 전달하는 단순하고 직접적인 언어를 사용해야 한다.

여기서 적절한 수준이란 두 가지 흔한 실패 사례 사이의 `Goldilocks` 지점을 의미한다.

한쪽 극단은 개발자가 원하는 에이전트의 행동을 정확하게 끌어내기 위해 복잡하고 깨지기 쉬운(brittle) 로직을 프롬프트에 하드코딩하는 경우다. 이 방식은 시스템을 취약하게 만들고, 시간이 지날수록 유지보수 복잡성을 높인다.

반대쪽 극단은 너무 추상적이고 포괄적인 지침만 제공하는 경우다. 이때 LLM은 원하는 결과를 생성하는 데 필요한 구체적인 신호를 받지 못하거나, 모델과 사람이 이미 같은 맥락을 공유하고 있다고 잘못 가정하게 된다.

가장 적절한 수준은 행동을 효과적으로 이끌 만큼 충분히 구체적이면서도, 동시에 모델이 스스로 판단할 수 있도록 강력한 휴리스틱을 제공할 만큼 충분히 유연한 상태다.

따라서 컨텍스트 엔지니어링에서는 시스템 프롬프트의 수준을 적절하게 조정(calibrate)하는 것이 중요하다.

참고 이미지: `Calibrating the system prompt`

이 이미지는 시스템 프롬프트를 조정할 때 발생하는 세 가지 상태를 보여준다.

- `Too specific`: 너무 구체적인 프롬프트는 수많은 조건문, intent 분기, 예외 규칙, 단계별 절차를 하드코딩한다. 겉보기에는 정밀해 보이지만 실제로는 brittle해지고, 새로운 상황에 약하며, 유지보수 비용이 커진다.
- `Too vague`: 너무 모호한 프롬프트는 "고객 문제를 브랜드 원칙에 맞게 해결하라"처럼 일반적인 원칙만 제공한다. 이 경우 모델은 실제 행동에 필요한 구체적인 신호를 받지 못해 일관된 결과를 내기 어렵다.
- `Just right`: 적절한 프롬프트는 역할, 사용 가능한 도구와 정책, 문제 해결 프레임워크, 판단 기준을 제공한다. 다만 모든 경우를 하드코딩하지 않고, 모델이 상황에 맞게 판단할 수 있는 휴리스틱을 남겨둔다.

따라서 `right altitude`는 구체성과 유연성 사이의 균형이다. 시스템 프롬프트는 모델이 해야 할 일을 충분히 명확히 알려주되, 모든 행동을 규칙으로 고정하지 않고 상황 판단이 가능하도록 설계해야 한다.

## 핵심 요약

컨텍스트 엔지니어링은 프롬프트를 잘 쓰는 문제를 넘어, 모델이 추론하는 순간에 어떤 정보가 컨텍스트 창에 들어가야 하는지를 설계하는 일이다.

- 프롬프트 엔지니어링: 모델에게 어떻게 지시할지 설계한다.
- 컨텍스트 엔지니어링: 모델이 어떤 정보, 도구, 기록, 외부 데이터, 메모리를 보고 판단하게 할지 설계한다.

에이전트는 여러 번의 추론과 도구 호출을 반복하기 때문에 매 턴마다 새로운 정보가 쌓인다. 이때 모든 것을 계속 컨텍스트에 넣으면 비용, 지연 시간, 혼란이 커진다. 그래서 필요한 것은 "가장 작은 고신호 토큰 집합"을 유지하는 것이다.

## 왜 중요한가

LLM의 컨텍스트 창은 커지고 있지만 무한하지 않다. 더 중요한 점은, 길게 넣을 수 있다고 해서 모델이 모든 정보를 똑같이 잘 활용하는 것은 아니라는 점이다.

- 컨텍스트가 길어질수록 중요한 정보를 놓치거나 혼동할 수 있다.
- 토큰은 단순 저장 공간이 아니라 attention budget을 소비하는 자원이다.
- transformer는 토큰 간 관계를 계산하므로, 컨텍스트가 길어질수록 모든 관계를 선명하게 잡기 어려워진다.
- 긴 컨텍스트는 hard limit 이전에도 성능 저하, 검색 정확도 저하, 장거리 추론 약화를 만들 수 있다.

따라서 좋은 에이전트는 "많이 넣는 시스템"이 아니라 "지금 판단에 필요한 것만 잘 넣는 시스템"이다.

## 효과적인 컨텍스트의 구조

### System prompt

시스템 프롬프트는 너무 구체적인 if-else 규칙 덩어리도, 너무 추상적인 원칙 모음도 아니어야 한다.

- 너무 낮은 altitude: 복잡한 로직을 프롬프트에 하드코딩해서 brittle해진다.
- 너무 높은 altitude: 구체적 신호가 부족해서 모델이 원하는 행동을 안정적으로 하지 못한다.
- 좋은 지점: 행동을 이끌 만큼 구체적이지만, 상황에 맞게 판단할 여지를 남긴다.

실무적으로는 `<background_information>`, `<instructions>`, `## Tool guidance`, `## Output description`처럼 섹션을 분리하는 방식이 도움이 된다. 다만 형식 자체보다 중요한 것은 기대 행동을 충분히 설명하면서도 불필요한 정보를 줄이는 것이다.

### Tools

도구는 에이전트가 외부 환경과 상호작용하고, 작업 수행 중 새로운 추가 컨텍스트를 가져올 수 있게 해 준다.

도구는 에이전트와 정보/행동 공간 사이의 인터페이스이자 계약이다. 따라서 도구는 토큰을 효율적으로 사용하는 정보를 반환해야 하며, 에이전트가 효율적으로 행동하도록 유도해야 한다.

도구 설계가 나쁘면 모델은 잘못된 도구를 고르거나, 같은 정보를 반복 조회하거나, 불필요한 결과를 컨텍스트에 쌓는다.

좋은 도구의 특징:

- 기능이 명확하고 서로 겹치지 않는다.
- 입력 파라미터가 모호하지 않다.
- 반환값이 토큰 효율적이다.
- 오류 상황에서도 모델이 다음 행동을 판단할 수 있게 해 준다.
- 인간도 어느 도구를 써야 할지 명확히 알 수 있다.

자주 보이는 실패 사례는 기능이 지나치게 많은 비대한 도구 집합이나, 어떤 도구를 사용해야 할지 판단하기 어려운 도구 구성이다. 사람 개발자조차 특정 상황에서 어떤 도구를 써야 하는지 명확히 판단할 수 없다면, AI 에이전트가 그보다 더 잘할 것이라고 기대해서는 안 된다.

도구가 많다고 좋은 에이전트가 되는 것이 아니다. 필요한 최소한의 도구 집합만 구성하면 장시간의 상호작용에서도 컨텍스트를 더 안정적으로 관리하고, 불필요한 정보를 줄일 수 있다. 반대로 애매하게 겹치는 도구 세트는 에이전트에게 선택 부담을 주고, 결국 컨텍스트 오염을 만든다.

### Examples

few-shot prompting은 잘 알려진 모범 사례이며, 여전히 강력하게 권장되는 방법이다. 하지만 가능한 모든 규칙과 예외 상황(edge case)을 프롬프트에 우겨 넣는 방식은 좋지 않다.

- 예시는 규칙 목록보다 행동 패턴을 더 잘 전달한다.
- 핵심은 edge case를 무한히 넣는 것이 아니라, 대표적이고 다양한 canonical example을 고르는 것이다.
- 예시는 모델에게 "이런 모양의 판단을 하라"는 그림을 제공한다.

좋은 예시는 LLM에게 "천 마디 말보다 값진 한 장의 그림"과 비슷한 역할을 한다. 시스템 프롬프트, 도구, 예시, 메시지 기록 등 컨텍스트를 구성하는 모든 요소의 핵심 원칙은 같다. 충분한 정보를 담되, 불필요하게 늘어지지 않도록 신중하게 관리해야 한다.

## Context retrieval과 agentic search

에이전트는 간단히 말해 "LLM이 도구를 반복적으로 사용하며 자율적으로 작업을 수행하는 것"으로 볼 수 있다. 기반 모델의 성능이 향상될수록 에이전트의 자율성도 함께 높아진다. 더 똑똑한 모델은 복잡하고 미묘한 문제를 스스로 해결하고, 오류가 발생해도 복구할 가능성이 커진다.

오늘날 많은 AI 네이티브 애플리케이션은 추론을 시작하기 전에 임베딩 기반 검색으로 필요한 컨텍스트를 미리 가져온다. 전통적인 RAG는 이런 방식에 가깝다. 하지만 업계가 에이전트 중심 접근으로 이동하면서, 검색 시스템에도 "just in time" 컨텍스트 전략을 함께 적용하는 경우가 늘고 있다.

just in time 방식:

- 전체 데이터를 미리 컨텍스트에 넣지 않는다.
- 파일 경로, 저장된 쿼리, 웹 링크, 식별자 같은 lightweight reference를 유지한다.
- 실행 시점(runtime)에 참조 정보를 이용해 도구로 필요한 데이터만 동적으로 불러온다.
- 큰 데이터는 `head`, `tail`, grep, 쿼리 같은 방식으로 부분적으로 탐색한다.

이 방식은 인간의 작업 방식과 비슷하다. 우리는 모든 문서를 외우지 않고, 파일 시스템/북마크/검색어/폴더 구조를 통해 필요할 때 찾아본다.

중요한 포인트는 metadata도 컨텍스트라는 점이다. lightweight reference는 저장 공간을 아끼는 역할만 하는 것이 아니라, 명시적이든 암묵적이든 에이전트의 행동을 조정하는 신호가 된다.

- 파일명
- 폴더 위치
- 최근 수정 시각
- 테스트 폴더인지 소스 폴더인지
- 링크 제목
- 쿼리 이름

이런 것들이 모델에게 "무엇을 먼저 봐야 하는지"에 대한 힌트를 준다. 예를 들어 파일 시스템에서 작업하는 에이전트에게 `tests/test_utils.py`와 `src/core_logic/test_utils.py`는 이름이 같아도 서로 다른 목적을 가진 파일로 해석된다.

## Progressive disclosure

에이전트가 스스로 탐색하고 데이터를 가져오게 하면 progressive disclosure가 가능해진다. 한 번에 모든 것을 아는 대신, 탐색하면서 점진적으로 필요한 컨텍스트를 열어보는 방식이다.

- 처음에는 가벼운 식별자와 구조만 본다.
- 필요한 파일이나 문서만 읽는다.
- 읽은 결과를 바탕으로 다음 탐색 대상을 정한다.
- 오래 유지해야 하는 정보는 노트나 요약으로 남긴다.

이 방식은 컨텍스트 창을 작업 메모리처럼 쓰게 한다. 모든 자료를 들고 있는 것이 아니라, 지금 판단에 필요한 부분만 들고 있는 것이다.

단점도 있다. 런타임 탐색은 미리 계산해 둔 데이터를 가져오는 방식보다 느릴 수 있다. 또한 LLM이 정보를 효과적으로 탐색할 수 있도록 적절한 도구와 휴리스틱을 갖추게 하려면 명확한 설계 철학과 신중한 엔지니어링이 필요하다.

가이드가 부족하면 에이전트는 도구를 잘못 사용하거나, 막다른 길을 계속 탐색하거나, 중요한 정보를 놓치면서 컨텍스트를 낭비할 수 있다.

그래서 현실적으로는 hybrid strategy가 유용하다.

- 속도를 위해 일부 데이터는 upfront로 미리 가져온다.
- 크거나 동적인 정보는 agentic search로 필요할 때 추가 탐색한다.
- 즉, 미리 가져오기와 runtime 탐색을 작업 환경에 맞게 섞는다.

Claude Code의 예시가 여기에 가깝다. `CLAUDE.md` 같은 규칙은 초기에 컨텍스트에 넣고, 실제 파일 내용은 glob/grep/read 같은 도구로 필요할 때 탐색한다.

## Long-horizon task를 위한 기법

긴 작업은 한 컨텍스트 창 안에서 끝나지 않는다. Long-horizon task에서는 전체 토큰 수가 LLM의 컨텍스트 윈도우를 초과하는 상황에서도, 에이전트가 일관성(coherence), 필요한 컨텍스트, 목표 지향적인 행동을 계속 유지해야 한다.

더 큰 컨텍스트 윈도우를 기다리는 것이 자연스러운 해결책처럼 보일 수 있다. 하지만 컨텍스트 윈도우가 커져도 `context pollution`과 `relevance` 문제는 계속 남을 가능성이 높다. 특히 최고의 에이전트 성능이 필요한 작업에서는, 단순히 더 많이 담는 것보다 무엇을 유지하고 무엇을 버릴지 결정하는 능력이 더 중요하다.

Anthropic은 에이전트가 장시간 작업을 효과적으로 수행하도록 컨텍스트 오염 문제를 직접 다루는 세 가지 기법을 제시한다.

- compaction
- structured note-taking
- multi-agent architectures

### 1. Compaction

Compaction은 컨텍스트 윈도우 한계에 가까워진 대화와 작업 상태를 요약한 뒤, 그 요약본을 바탕으로 새로운 컨텍스트 윈도우를 시작하는 기법이다.

좋은 compaction은 단순 압축이 아니라 중요한 상태 보존이다. 핵심은 무엇을 남기고 무엇을 버릴지를 잘 선택하는 것이다.

- 유지할 것: 아키텍처 결정, unresolved bug, 구현 세부사항, 현재 목표, 최근 접근 파일
- 버릴 것: 오래된 raw tool output, 반복된 메시지, 이미 환경에 남아 있는 정보

너무 공격적으로 요약하면 나중에 중요성이 드러나는 미묘하지만 핵심적인 컨텍스트를 잃을 수 있다. 그래서 compaction prompt는 복잡한 agent trace를 기반으로 세심하게 튜닝해야 한다.

튜닝할 때는 처음부터 짧고 깔끔한 요약을 목표로 하기보다, 먼저 recall을 최대화해 실행 기록의 중요한 정보를 최대한 빠짐없이 담도록 만드는 것이 좋다. 이후 불필요한 내용을 제거하면서 precision을 높여간다.

낮은 위험의 compaction으로는 tool result clearing이 있다. 오래된 도구 호출 결과는 다시 필요하면 도구로 재조회할 수 있으므로, raw result를 계속 들고 있을 필요가 적다.

### 2. Structured note-taking

Structured note-taking, 또는 agentic memory는 에이전트가 컨텍스트 윈도우 밖의 저장소에 주기적으로 메모를 기록하는 기법이다. 필요할 때 이 메모들을 다시 컨텍스트 윈도우로 불러와 활용한다.

즉, 모든 작업 기록을 컨텍스트 안에 계속 들고 있는 대신, 오래 유지해야 하는 상태를 외부 저장소에 구조화해서 남기는 방식이다.

예:

- TODO 목록
- NOTES.md
- 진행 요약
- 결정 기록
- 다음 세션을 위한 handoff

이 방식은 컨텍스트 창이 초기화되어도 작업의 continuity를 유지하게 해 준다.

### 3. Sub-agent architectures

Sub-agent architecture는 하나의 에이전트가 프로젝트 전체의 상태와 모든 세부 컨텍스트를 유지하려고 하지 않는 구조다. 대신 각각의 전문화된 서브 에이전트가 독립적이고 깨끗한 컨텍스트 윈도우에서 특정 작업을 담당한다.

메인 에이전트는 전체적인 계획을 조율하고, 서브 에이전트들은 심층적인 기술 작업을 수행하거나 필요한 정보를 찾기 위해 다양한 도구를 사용한다. 이후 서브 에이전트는 조사/분석 결과의 요약만 메인 에이전트에게 반환한다.

장점:

- 세부 탐색 컨텍스트가 메인 에이전트를 오염시키지 않는다.
- 병렬 조사에 적합하다.
- 메인 에이전트는 종합과 판단에 집중할 수 있다.

작업 특성별 선택:

- compaction: 긴 대화 흐름과 연속성이 중요한 작업
- structured note-taking: 반복 개발, 명확한 milestone이 있는 작업
- sub-agent: 복잡한 리서치, 병렬 탐색이 이득인 작업

## Conclusion

컨텍스트 엔지니어링은 LLM을 활용해 시스템을 만드는 방식 자체의 근본적인 변화를 의미한다.

모델의 성능이 높아질수록 과제는 단순히 완벽한 프롬프트를 작성하는 것이 아니라, 각 단계에서 모델의 제한된 컨텍스트와 attention 공간에 어떤 정보를 넣을지 신중하게 선별하는 것이 된다.

장기 작업을 위해 컨텍스트를 압축하든, 토큰 효율적인 도구를 설계하든, 에이전트가 필요한 순간에 환경을 탐색하도록 만들든, 핵심 원칙은 동일하다.

원하는 결과가 나올 가능성을 최대화할 수 있는 가장 작지만 정보량이 높은(high-signal) 토큰 집합을 찾는 것이다.

이미 더 뛰어난 모델일수록 세세한 지시를 덜 필요로 하며, 그만큼 에이전트가 더 높은 자율성을 가지고 동작할 수 있다는 점이 나타나고 있다.

하지만 모델 성능이 계속 향상되더라도, 컨텍스트를 소중하면서도 한정된 자원으로 다루는 것은 신뢰할 수 있고 효과적인 에이전트를 만드는 핵심 원칙으로 남을 것이다.

## 내 생각

이 글은 "컨텍스트를 많이 넣으면 좋다"는 직관을 정면으로 꺾는다. 에이전트 설계에서 중요한 것은 max context window 크기가 아니라, 매 순간 모델의 attention을 어디에 쓰게 할지다.

특히 지금 학습 중인 LangGraph/RAG/agent 흐름과 연결하면:

- RAG는 "검색해서 넣기"가 아니라 "언제, 무엇을, 얼마나 넣을지 결정하는 정책"까지 포함해야 한다.
- LangGraph의 state는 무조건 많이 들고 다니는 저장소가 아니라, 현재 노드 판단에 필요한 working memory로 설계해야 한다.
- tool output은 그대로 누적하면 위험하다. 요약하거나, 식별자만 남기거나, 필요할 때 재조회하는 방식이 낫다.
- handoff 문서와 학습 노트는 structured note-taking의 실제 예시로 볼 수 있다.

## 실습 아이디어

현재 `raw/light study/langchain_quickstart.py`나 `raw/graph_rag_tutorial/2026_06_22/rag_vectorstore.py`를 바탕으로 작은 실습을 해볼 수 있다.

1. 같은 질문에 대해 raw document chunk를 많이 넣는 버전과 top-k를 줄인 버전을 비교한다.
2. tool output 전체를 state에 누적하는 버전과 요약/ID만 남기는 버전을 비교한다.
3. LangGraph state에 `messages`, `notes`, `retrieved_refs`, `final_context`를 분리해 본다.
4. 각 단계에서 "지금 모델이 꼭 알아야 하는 정보"와 "필요하면 다시 찾을 수 있는 정보"를 나눠 본다.

## 기억할 문장

좋은 에이전트는 모든 정보를 들고 있는 에이전트가 아니라, 필요한 정보를 필요한 순간에 가져오고 오래 유지해야 할 상태만 구조화해서 남기는 에이전트다.
