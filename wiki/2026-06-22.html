임베딩과 벡터 유사도 검색

GraphRAG를 다루기 전에, 기존 벡터 기반 RAG가 어떻게 동작하고 어디서 한계를 보이는지부터 확인했다. 출발점은 임베딩이다. 텍스트를 OpenAIEmbeddings(text-embedding-3-small)로 변환하면 의미가 담긴 고차원 벡터(1536차원)가 나오고, 두 벡터의 코사인 유사도로 "의미가 얼마나 비슷한지"를 수치화할 수 있다.

  • 코사인 유사도는 (v1 · v2) / (|v1| * |v2|)로 계산한다 — 벡터의 방향(의미)이 얼마나 일치하는지를 보고, 크기(텍스트 길이 등)는 무시한다.
  • 날씨 관련 문장들끼리는 유사도가 높고, 프로그래밍 관련 문장과는 유사도가 낮게 나오는 것을 직접 확인했다.
  • 벡터스토어(InMemoryVectorStore)를 쓰면 이 유사도 계산과 정렬(top-k 추출)을 직접 구현할 필요 없이 similarity_search 한 줄로 처리된다.
  • 흥미로운 점: 한국어 문서 집합에서 영어 질문("Who makes Korean Language")을 던져도 관련 문서가 검색됐다 — 임베딩 모델이 언어를 넘어 의미 공간에서 매칭하기 때문.
graph LR q["질문/문서 텍스트"] --> e["임베딩 모델
(text-embedding-3-small)"] e --> v["벡터"] v -->|코사인 유사도| s["유사도 점수"] s -->|상위 k개| r["검색 결과"]
코사인 유사도 직접 계산하는 코드 
def cosine_similarity(vec1, vec2):
    vec1 = np.array(vec1)
    vec2 = np.array(vec2)
    return np.dot(vec1, vec2) / (np.linalg.norm(vec1) * np.linalg.norm(vec2))

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
doc_vectors = embeddings.embed_documents(documents)

def search(query: str, top_k: int = 3) -> list:
    query_vector = embeddings.embed_query(query)
    similarities = [(i, cosine_similarity(query_vector, v)) for i, v in enumerate(doc_vectors)]
    similarities.sort(key=lambda x: x[1], reverse=True)
    return [{"document": documents[i], "similarity": s} for i, s in similarities[:top_k]]

텍스트 청킹

긴 문서를 그대로 임베딩하면 의미가 희석되므로, 임베딩 전에 적절한 크기로 잘라야 한다. RecursiveCharacterTextSplitter는 chunk_size(청크 최대 길이)와 chunk_overlap(청크 간 겹치는 길이)을 지정해 문단·문장 단위를 최대한 보존하면서 분할한다. 겹치는 구간을 두는 이유는 청크 경계에서 문맥이 끊기는 것을 완화하기 위함이다.

벡터 검색 기반 RAG와 그 한계

세종대왕, 정조, 이순신, 집현전에 대한 한국사 문서 6개를 청킹 → 임베딩 → 벡터스토어 저장 → retriever + LLM 체인으로 묶어 표준적인 RAG 파이프라인을 구성했다.

graph LR docs["문서들"] --> split["청킹"] split --> embed["임베딩"] embed --> store["벡터 스토어"] q["질문"] --> retriever["retriever
(top-k 검색)"] store --> retriever retriever --> ctx["검색된 컨텍스트"] ctx --> prompt["프롬프트"] q --> prompt prompt --> llm["LLM"] llm --> ans["답변"]

단순 사실 조회 질문("세종대왕이 발명한 것들은?")에는 잘 답했지만, 의도적으로 어려운 질문을 던져보니 한계가 분명했다.

  • 관계 추론: "세종대왕과 집현전의 관계는?" — 두 개체 사이의 연결고리를 벡터 유사도만으로는 명시적으로 찾기 어렵다.
  • 다단계 추론: "세종대왕이 설치한 기관에서 누가 활동했나?" — "세종대왕 → 집현전 설치 → 집현전에서 활동한 인물"처럼 두 단계를 거쳐야 하는데, 벡터 검색은 질문 자체와 유사한 청크만 가져오기 때문에 중간 연결 단계를 놓치기 쉽다.
  • 비교 질문: "세종대왕과 정조의 공통점은?" — 두 인물에 대한 청크를 각각 가져와도, 그 안에서 공통점을 뽑아내는 건 검색이 아니라 LLM의 추론에 전적으로 의존하게 된다.
  • 시간순 나열: "조선의 왕들을 시대순으로 나열" — 문서에 시간 순서가 구조화되어 있지 않으면 검색만으로는 답할 수 없다.

즉, 벡터 검색은 "질문과 의미적으로 비슷한 텍스트 조각"을 찾는 데는 강하지만, 개체(entity) 사이의 관계나 다단계 연결을 명시적으로 표현하지 못한다. 이 한계가 바로 GraphRAG의 동기다 — 문서를 청크가 아니라 개체와 관계로 이루어진 지식 그래프로 표현하면, "세종대왕 -[설치]-> 집현전 -[소속]-> 성삼문" 같은 경로를 그래프 탐색으로 직접 따라갈 수 있다.

Neo4j로 그래프 데이터베이스 다루기

GraphRAG의 그래프 저장소로 쓰이는 Neo4j를 처음 연결해보고, 기본적인 노드/관계 생성과 조회를 익혔다.

  • GraphDatabase.driver(URI, auth=...)로 드라이버를 만들고 verify_connectivity()로 연결을 확인한다.
  • Cypher 쿼리로 노드(CREATE (p:Person {...}))와 그 사이의 관계(CREATE (p)-[:WORKS_AT]->(c))를 한 번에 생성할 수 있다.
  • 조회도 패턴 매칭으로 한다: MATCH (p:Person)-[:WORKS_AT]->(c:Company) RETURN ... — "관계를 따라가는 질의"가 SQL의 JOIN보다 직접적으로 표현된다.
graph LR p["Person
김철수"] -->|WORKS_AT| c["Company
AI 스타트업"]
노드/관계 생성 및 조회 코드 
with driver.session() as session:
    session.run("""
        CREATE (p: Person {name: '김철수', age: 30})
        CREATE (c: Company {name: 'AI 스타트업'})
        CREATE (p)-[:WORKS_AT]->(c)
    """)

    result = session.run("""
        MATCH (p:Person)-[:WORKS_AT]->(c:Company)
        RETURN p.name as person, c.name as company
    """)
    for record in result:
        print(f"{record['person']}님은 {record['company']}에서 일합니다.")

이 노드-관계 모델이 다음 단계의 핵심이다. 위에서 확인한 "다단계 추론", "관계 질문" 같은 한계를, 문서에서 추출한 개체와 관계를 Neo4j 그래프로 저장해두고 그래프 탐색으로 보완하는 것이 GraphRAG의 방향이다.

언어 모델이 챗GPT가 되기까지: 스케일링과 정렬

모델 크기와 학습 데이터가 커질수록 성능이 좋아지는 이유를, 학습 과정 자체를 데이터 압축으로 보는 관점에서 다시 짚었다. 언어 모델은 학습 데이터를 그대로 외우는 게 아니라, 데이터 안에서 공통되고 중요한 패턴만 남기는 손실 압축을 수행한다고 볼 수 있다. 이 관점에서는 학습 데이터의 크기가 모델이 의미 있게 키울 수 있는 최대 크기의 상한이 된다 — 데이터가 적은데 모델만 키우면 압축할 패턴이 부족해 과적합으로 이어지기 쉽다.

GPT-3 같은 사전 학습 모델은 "다음 단어 예측"만 학습했을 뿐, 사용자의 요청 의도에 맞게 답하도록 훈련되지는 않았다. LLM의 답변을 사용자 의도에 맞추는 과정을 정렬(alignment)이라고 부르고, GPT-3를 챗GPT로 바꾼 것이 바로 이 정렬 작업이다.

graph LR pretrain["사전 학습
(다음 단어 예측)"] --> sft["지도 미세 조정(SFT)
지시 데이터셋으로 추가 학습"] sft --> rlhf["RLHF
사람 피드백 기반 강화학습"] rlhf --> aligned["정렬된 모델
(예: ChatGPT)"]
  • 지도 미세 조정(SFT): 정렬의 핵심 과정. 사용자의 요청·지시와 그에 대한 적절한 응답을 정리한 지시 데이터셋으로 모델을 추가 학습시킨다.
  • RLHF: 사람의 피드백(어떤 답변이 더 나은지에 대한 선호)을 보상 신호로 활용해 모델을 강화 학습시키는 단계. SFT만으로 다 잡지 못하는 답변의 품질·안전성을 추가로 조정한다.
